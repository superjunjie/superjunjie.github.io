微前端架构

Created2022-10-11|Updated2022-12-09|微前端
|Post View:

背景

微前端是由微服务架构演化而来，微前端的提出，主要是为了解决以下问题：

  1. 前端项目越做越大，构建速度越来越慢
  2. 团队开发人员很多，产品功能复杂，牵一发而动全身
  3. 代码冲突频繁，影响面大
  4. 内心想做sass产品，客户想要定制化

什么是微前端

微前端(opens new window)由ThoughtWorks 2016年提出，将后端微服务的理念应用于浏览器端，即将 Web 应用由单一的单体应用转变为多个小型前端应用聚合为一的应用
微前端是一种架构风格，其中众多独立交付的前端应用组合成一个大型整体
微前端是一种多个团队通过独立发布功能的方式来共同构建现代化 web 应用的技术手段及方法策略。

核心价值

  1. 技术栈无关。主框架不限制接入应用的技术栈，微应用具备完全自主权
  2. 独立开发、独立部署。微应用仓库独立，前后端可独立开发，部署完成后主框架自动完成同步更新
  3. 增量升级。在面对各种复杂场景时，我们通常很难对一个已经存在的系统做全量的技术栈升级或重构，而微前端是一种非常好的实施渐进式重构的手段和策略
  4. 独立运行。每个微应用之间状态隔离，运行时状态不共享

Portal App

说明

是一种基于基座+路由分发实现的单技术栈微前端架构。主要有以下三个部分组成：

  1. FrameWork（框架相关代码，通过git subtree引入到每个APP中。framework包括子应用注册逻辑，webpack配置项、全局utilis方法、全局router/store/i18n/http等）
  2. Protal App（基座APP，包含网站公共页面（不属于某个业务），如login page/login-terms等页面）
  3. Micro App（基于业务拆分出来的子应用，包含各个业务的前端页面）

架构图

核心价值

  1. 独立开发、独立部署。微应用仓库独立，前后端可独立开发，部署，使用github action构建之后发到AWS S3上主框架自动完成同步更新
  2. 多版本/AB环境发布。结合AWS S3和子应用动态加载（路由守卫+配置中心）实现
  3. 公共代码分离。
  4. 通过配置动态生成topMenu和sideBar。

构建发布

独立发布

引入framework代码构建子应用，打包出来的代码包含framework全部功能，可作为独立应用发布到指定环境。其中portal App以及在本地部署的时候使用的就是这种方式

作为子应用发布

作为主应用的子应用构建，不引入framework代码，构建出来的产物不包含framework代码。原理通过使用主应用挂载到window上面的注册方法将自身的路由注册到主应用实现

入口方法

Entry是一个类方法，它主要做了下面这些事情：

  1. 提供子应用注册router、vuex、i18n的注册方法
  2. 将vue、router、store等全局方法挂载到global(默认是window)上
  3. 设置cookie，用于非生产环境，做子应用切换时重置cookie
  4. 获取网站的配置信息，如cdnUrl、microAppData、用户登录信息等
  5. 将子应用配置信息注册到主应用中
  6. 设置全局路由守卫，在路由信息变更时，判断跳转地址是不是当前微应用，如果不是就loadScript跳转后的子应用
  7. 设置有关权限路由的路由守卫，判断页面是否由权限，如果没有就跳转到not-permission页面
  8. 设置网站的配置信息，如网站的theme、favicon、favicon、mainColor
  9. 创建主应用实例

应用注册机制

注册Router

使用addRouters将子应用路由注册到主应用中。addRouters接受三个参数，第一个是子应用的名称，第二个是子应用路由前缀，第三个是要注册的路由信息。在这里如果判断prefix为null则表明是Portal App，不需要使用前缀，对于其他Micro App则需要遍历routes并加入前缀信息，最后使用VueRouter提供的原生方法addRoutes将子应用路由信息加入到主路由

注册Vuex

使用Vuex提供的registerModule将子应用的store注册到主应用中

注册i18n

使用mergeLocaleMessage api注册多语言信息

动态加载及版本控制

主应用包含多个子应用，在加载网站的时候，不需要一次把所有的子应用都加载进来，只需要加载当前被激活的子应用即可。这样以来，首次加载出来的页面只占整个应用的一小部分，网站在响应和渲染上就会有显著性能提升。
在版本控制方面，由于网站时动态加载的，所有在获取子应用资源时，可以做子应用的版本控制，这对sass网站显得尤为重要。
作为一个sass网站，通常要面对很多客户接入，客户之间又是众口难调，如何做一个通用的sass平台，兼容所有客户的各种定制化需求，这对网站开发者来说是一个巨大挑战。如果将各个客户的业务代码和定制化需求写在一起，通过配置的方式去实现，那么当接入的客户越来越多时，项目维护起来肯定会越来越复杂，一个小的改动，都可能会导致整个系统出现不可预知的错误。
对此实现对网站的动态配置就显得尤为重要，在面对客户的定制化需求时，从主开发线上为客户切一个分支出来，然后在这个支线上满足其定制化需求，就容易多了，同时客户也可以选择是否接入新上线的功能，这样便满足了客服的定制化需求，在有新的客户接入时，便可以以最快的速度，最少的成本满足客服的需求。

动态加载流程

  1. 全局路由守卫，当路由发生变化，执行脚本
  2. 判断要前往的路由是否已加入主应用，是就直接前往该页面
  3. 根据配置获取子应用资源，并通过script引入子应用的资源
  4. 执行子应用的资源，将子应用加入到子应用中
  5. 完成对子应用的加载，并跳转到对应页面 动态脚本加载方法
